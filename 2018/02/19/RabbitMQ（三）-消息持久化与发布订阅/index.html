RabbitMQ（三）-消息持久化与发布订阅

发表于 2018-02-19 | | 阅读次数:

上一次我们只是简单的接收了消息，此次，我们将创建一个Work Queue–工作队列，Producer分配任务给多个Worker，同时不立即执行资源密集型任务，避免Worker不必要的等待。

此时的任务和之前提到的消息是等价的，在后台运行的工作进程将弹出的任务，并最终执行作业。当运行许多Worker时，任务将在他们之间共享。

消息确认（防丢失）

概念

一个任务的完成通常只需要很短的时间。但当考虑到，可能某个消费者搞了个又大又长的任务来，然后他只完成了一部分，你就把他kill了，或者说自己就挂了。

如果此时我们选择不作为，消息（任务）就丢失了，那最糟的情况下可能整个大任务也over，消息失效。

所以此时我们是不是应该做些什么？最直观的想法就是，这个worker干不了，我们交给新的worker来干。让我们来看看下面这段文字，了解下RabbitMQ的机制：

In order to make sure a message is never lost, RabbitMQ supports message acknowledgments. An ack(nowledgement) is sent back by the consumer to tell RabbitMQ that a particular message has been received, processed and that RabbitMQ is free to delete it.

If a consumer dies (its channel is closed, connection is closed, or TCP connection is lost) without sending an ack, RabbitMQ will understand that a message wasn’t processed fully and will re-queue it. If there are other consumers online at the same time, it will then quickly redeliver it to another consumer. That way you can be sure that no message is lost, even if the workers occasionally die.

从上面官方提供的解释，我们可以看出RabbitMQ和常见协议是一样的，只有当接收者(consumer)回复了一个确认收到的ack，RabbitMQ才能删除这个task。当然，如果worker挂了，没有在指定时间内返回ack(ack超时)，RabbitMQ有两种做法：

  1. 如果没有空闲的consumer,那么re-queue 重新分配；
  2. 反之，此时有空闲的consumer，RabbitMQ会迅速把消息（任务）交给某个空闲的。

发布订阅

前面我们只是简单的实现了“一对一模型”，了解了RabbitMQ的一些持久化方案，现在我们要实现正真意义上的“publish/subscribe”，一个消息可以被发送给多个消费者。

通过一个简单的日志系统来说明，消息生产者会将日志发送给队列，然后多个订阅者可以接收到这条日志。

Exchanges

RabbitMQ中，消息生产者从来不会将消息直接发送给某个队列，甚至有时候都不知道这个消息会不会发送到队列中去。

生产者只能发送消息到交换机(exchange)。它的一端接收来自producers的消息，另一端发送消息给queues。同时，它也必须知道处理这个消息的规则，比如发送给某个特定队列，发送给多个队列，或者丢弃它。这些规则是通过exchange type来定义。
